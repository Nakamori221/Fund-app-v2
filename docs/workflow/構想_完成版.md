- **スケジュール**：**2025-10-27〜2026-02-22**で Phase 0〜5 を推進（意思決定→実装→ダミー検証→本番移行）。fileciteturn1file1

## 1. 背景と目的
### 背景
- 各部門で**定義/フォーマットが不統一** → 集計・再計算の手戻りが多発。
- レビュー/承認の履歴が**分散** → 監査性が低い。
- LP向け**開示ルールのばらつき** → 手戻りコスト/コンプライアンス・リスク。fileciteturn1file0

### 目的
- IC資料作成の**標準化と自動化**により、**作業時間の削減**と**品質の安定化**、**監査可能性の向上**を同時達成。fileciteturn1file0

---

## 2. 到達像（Step 5：本番自動運用の“日常”）
- **流れ**：収集（PUB/EXT）→ 正規化 → 指標再計算 → 要約生成 → 開示判定/マスキング → 承認 → 配信 → 監査ログまで**イベント連鎖で自動化**。fileciteturn1file0
- **人の役割**：例外処理と意思決定に専念（ハイリスク項目のみレビュー強制）。fileciteturn1file0
- **管理**：SLA/監視ダッシュボード、異常検知・再実行・ロールバックを標準化。fileciteturn1file0

### In（初期）

## 7. アーキテクチャと運用
- **スタック**：FastAPI / PostgreSQL / Redis / React / （Cloud Run or K8s）/ GitHub Actions。収集に Playwright、要約に LLM（OpenAI/Vertex）。監視に Datadog 等。fileciteturn1file3
- **データフロー**：PUB/EXT → ETL正規化 → KPI再計算 → 要約/生成 → 開示ルール → 承認 → 配信 → 監査ログ。fileciteturn1file0
- **運用**：隔週ステコミ、週次進捗・リスク、月次品質/予算レビュー、Phase終端のデモ/振返り。fileciteturn1file16

## 8. セキュリティ・開示統制・監査
- **開示レベル**：`IC/LP/LP+NDA/Private × sensitivity` を**自動判定**し、**マスキング/アクセス**を適用。適用履歴は監査ログに**完全記録**。fileciteturn1file6
- **チェック**：`source_tag` と `evidence` の突合、例外は二重承認。fileciteturn1file6
- **フォールバック**：誤適用時はロールバック/エスカレーションの標準手順。fileciteturn1file1
