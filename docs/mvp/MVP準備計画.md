- **次アクション**: Phase 1（v1）の実データ入力・DB連携・レポート運用フロー実装へ移行。

## 作業ブレークダウン
### Phase 1 (v1): 定型化・DB格納・プレビュー生成
- **P1-1 スキーマ設計**: 観測テーブル（source_tag, disclosure, as_of, confidence, status 等）とギャップテーブル、資料テンプレ（IC/LP）のフィールド定義を文章化。
- **P1-2 既存資料収集とタグ付け**: 手元資料のリスト化、タグ付け基準（PUB/EXT/ANL/CONF/INT）とメタデータ取得方法の決定。
- **P1-3 データ入力プロセス整備**: 半手動インジェストスクリプト／ノーコード入力フォームの選定、操作手順のドラフト作成。
- **P1-4 レポートプレビュー実装**: 欠損を空欄または `status: pending` として表示するMarkdown出力、章ごとの完了率メタ情報表示。
- **P1-5 品質チェック初期セット**: 出典必須チェック、As-of未入力検知、ログ出力仕様の作成。

### Phase 1.5: ファクトチェック強化
- **P1.5-1 ソース重複検知**: 同一主張への複数ソース突合ロジックと優先順位ルール（一次＞二次）を実装。
- **P1.5-2 差異ハンドリング**: 乖離時のフラグ付け／レビューキュー作成、承認担当者とSLA定義。
- **P1.5-3 証跡管理**: 検証ログ（検証日・検証者・結果・備考）をDBへ保存、レポート脚注と連携。

### Phase 2: CONFフォルダ連携
- **P2-1 アクセスコントロール設計**: 権限マトリクス、監査ログ要件、鍵管理ポリシーの確定。
- **P2-2 抽出フロー実装**: CONFファイルから必要項目のみ抽出し要約テキスト化、LLM非依存の前処理を構築。
- **P2-3 承認UIモック**: 抽出結果を人が確認承認する最小UI（CLI／スプレッドシート／軽量Web）の試作。
- **P2-4 レポート連携**: CONF情報をIC/LP両方へ反映しつつ、LP向けマスキング規則を適用するロジック追加。

### Phase 3: INT連携
- **P3-1 インタビューフォーマット定義**: 質問ID、役職ターゲット、記入フォーマット（テキスト/数値）、同意チェック項目のテンプレ化。
- **P3-2 収集プロセス整備**: インタビュワー用ガイドと入力フォーム提供、提出→レビュー→DB登録の流れを設定。
- **P3-3 INTデータ統合**: 観測テーブルへのマッピング、未解決ギャップの自動更新、レポート差分ハイライト。
- **P3-4 最終完成チェック**: 全セクションの完了率100%時の最終出力テンプレと承認プロセス確立。

## 横断タスク
- **ログ・監査**: すべての収集・更新操作に対してタイムスタンプ／操作者／ソースを記録する仕組みを構築。
- **運用ガイドライン**: Web検索の利用範囲、CONF取扱い、INT録音・テキスト化の同意手順を文書化。
- **セキュリティチェックリスト**: アクセス権レビュー、鍵のローテーション、データマスキングの検証などを定期的に実施。
- **テストデータ準備**: SaaS/Marketplace/Fintechのダミー案件3件を整備し、各フェーズで回帰テスト可能にする。

## 役割と責任（ドラフト）
- **プロダクトオーナー**: フェーズ優先順位決定、承認フロー最終決裁。
- **データエンジニア**: スキーマ設計、DB実装、抽出スクリプト整備。
- **リサーチャー**: Web検索での情報収集、ファクトチェック、反証ログ管理。
- **投資担当（IC責任者）**: CONF承認、最終レポートレビュー。
- **インタビュワー**: Phase 3での質問実施、回答フォーマット入力。

## リスクと対策
- **ソース不足**: Web検索だけで裏付けが取れない場合、早期にCONF/INT依存箇所を明示して優先順位調整。
- **権限調整遅延**: CONFフォルダアクセス許可が遅れる場合の暫定運用（手動アップロード）を定義。
- **データ品質ばらつき**: タグ付け基準のドキュメント化と定期レビューを実施。
- **スケジュール超過**: マイルストーンごとに進捗レビューを行い、必要に応じてスコープ調整。
