# ベンダー選定ガイド（外部委託検討用）

## 1. 内製 vs 外部委託の判断基準

### 意思決定マトリックス

| 要因 | 内製を推奨 | 外部委託を推奨 |
|------|----------|--------------|
| **技術リソース** | エンジニア2-3名確保可能 | エンジニア確保困難 |
| **予算** | 長期的投資が可能（3年+） | 初期コスト重視 |
| **カスタマイズ** | 独自プロセスが多い | 標準的なプロセス |
| **セキュリティ** | 機密情報を外部に出せない | 標準的なセキュリティで可 |
| **スピード** | 6-12ヶ月の開発期間OK | 即座に必要 |
| **運用体制** | 社内で運用・保守可能 | 運用リソースなし |
| **差別化** | システムで競争優位性を得たい | ツールは手段に過ぎない |

### 推奨判断

```yaml
✓ 内製を強く推奨:
  - 年間案件数 > 50件
  - エンジニアリング体制あり
  - 独自のプロセス・ノウハウがある
  - 長期的な競争優位性を重視

✓ 外部委託を検討:
  - 年間案件数 < 20件
  - 技術リソースが限定的
  - 早期のROI実現を重視
  - 標準的なプロセスで十分

✓ ハイブリッド（部分委託）:
  - 開発は外部、運用は内製
  - Phase 1のみ外部、Phase 2-5は内製
  - コアロジックは内製、周辺は外部
```

## 2. ベンダータイプ別の特徴

### Type A: 総合SIer（大手）

```yaml
特徴:
  - 会社規模: 大（従業員1000名以上）
  - 実績: 豊富（金融機関等）
  - 価格: 高（¥50M-100M+）
  - 納期: 長い（12-18ヶ月）

強み:
  ✓ 大規模システムの実績
  ✓ セキュリティ・コンプライアンス対応が万全
  ✓ 長期的なサポート体制
  ✓ 金融業界の知見

弱み:
  ✗ 柔軟性が低い
  ✗ コストが高い
  ✗ 最新技術への対応が遅い
  ✗ スタートアップのスピード感に合わない

適合ケース:
  - 大規模ファンド（AUM 1000億円+）
  - 規制対応が最優先
  - 予算が潤沢
  - 失敗が許されない

主なベンダー例:
  - NTTデータ
  - 野村総合研究所（NRI）
  - 富士通
  - アクセンチュア
```

### Type B: 中堅ソフトウェア会社

```yaml
特徴:
  - 会社規模: 中（従業員50-500名）
  - 実績: あり（業界特化）
  - 価格: 中（¥20M-50M）
  - 納期: 中（6-12ヶ月）

強み:
  ✓ バランスが良い（品質・コスト・スピード）
  ✓ 業界特化型の知見
  ✓ 柔軟な対応
  ✓ 専任チームの確保

弱み:
  ✗ 最先端技術への対応は限定的
  ✗ リソースの制約
  ✗ グローバル対応は限定的

適合ケース:
  - 中規模ファンド（AUM 100-1000億円）
  - バランス重視
  - 業界特化の機能が必要

主なベンダー例:
  - TIS
  - SCSK
  - 日立ソリューションズ
  - ビジネスエンジニアリング
```

### Type C: AI/LLM特化スタートアップ

```yaml
特徴:
  - 会社規模: 小（従業員10-50名）
  - 実績: 限定的だが最新技術
  - 価格: 低-中（¥10M-30M）
  - 納期: 短い（3-6ヶ月）

強み:
  ✓ 最新のAI技術
  ✓ スピードが速い
  ✓ 柔軟性が高い
  ✓ コストが比較的安価
  ✓ スタートアップのニーズを理解

弱み:
  ✗ 実績が少ない
  ✗ 会社の存続リスク
  ✗ 大規模対応の経験不足
  ✗ セキュリティ体制が不十分な場合も

適合ケース:
  - 小-中規模ファンド（AUM <500億円）
  - AI技術を重視
  - 早期のリリースを優先
  - リスクを取れる

主なベンダー例:
  - 各種AIスタートアップ
  - システム開発会社のAI部門
```

### Type D: フリーランス/小規模チーム

```yaml
特徴:
  - 会社規模: 個人-5名程度
  - 実績: 個人の実績次第
  - 価格: 低（¥5M-15M）
  - 納期: 短-中（2-8ヶ月）

強み:
  ✓ コストが最も低い
  ✓ 柔軟性が最も高い
  ✓ 直接コミュニケーション
  ✓ スピードが速い場合も

弱み:
  ✗ リソースの制約（人数）
  ✗ 継続性の不安
  ✗ サポート体制が弱い
  ✗ 複雑なプロジェクトは困難

適合ケース:
  - 小規模ファンド（AUM <100億円）
  - 予算制約が厳しい
  - MVP/パイロットから始めたい
  - 社内にプロジェクト管理能力あり

探し方:
  - Lancers, CrowdWorks
  - ビズリーチ
  - Wantedly
  - GitHub, Qiita経由
```

## 3. ベンダー評価チェックリスト

### Step 1: 事前評価（書類審査）

```markdown
□ 会社情報
  □ 設立年・資本金・従業員数
  □ 財務状況（上場企業の場合）
  □ 主要な取引先
  □ 離職率・平均勤続年数

□ 技術力
  □ 使用技術スタック（要一致）
  □ LLM統合の実績
  □ GitHub公開実績（あれば）
  □ 技術ブログ・カンファレンス登壇

□ 実績
  □ 類似プロジェクトの経験
  □ 金融・VC業界の経験
  □ AIシステムの開発実績
  □ 成功事例・失敗事例

□ セキュリティ
  □ ISO 27001等の認証
  □ プライバシーマーク
  □ セキュリティ監査の実施状況
  □ インシデント履歴

□ 提案内容
  □ 要件理解の深さ
  □ 提案の具体性
  □ 想定される課題への言及
  □ スケジュール・体制の妥当性
```

```markdown

## 2. 要件

（詳細は別紙「要件定義書」参照）

## 3. 制約条件
```

## 4. 契約のポイント

### 契約形態の選択

```yaml
【請負契約】
特徴:
  - 成果物ベース
  - 固定価格
  - ベンダーがリスクを負う

メリット:
  ✓ 予算が明確
  ✓ 納期が確定
  ✓ 要件が固まっている場合に最適

デメリット:
  ✗ 仕様変更が困難（追加費用）
  ✗ 要件定義が不十分だとトラブル

推奨ケース:
  - 要件が明確
  - 大規模ベンダー
  - Phase 1のみ

【準委任契約】
特徴:
  - 時間ベース
  - 柔軟な対応
  - 顧客がリスクを負う

メリット:
  ✓ 仕様変更が容易
  ✓ アジャイル開発に適合
  ✓ 不確実性が高い場合に最適

デメリット:
  ✗ 予算が不明確
  ✗ 納期が不確定
  ✗ 管理工数が増える

推奨ケース:
  - 要件が流動的
  - スタートアップベンダー
  - Phase 2以降
```

### 重要な契約条項

```markdown
□ 知的財産権
  - ソースコードの著作権は顧客に帰属
  - ベンダーの汎用ライブラリは除く
  - 第三者ライブラリのライセンス確認

□ 秘密保持
  - NDAの締結（相互）
  - 機密情報の定義
  - 契約終了後の義務継続

□ 瑕疵担保
  - 期間: 納品後1年間
  - 対象: 重大なバグ・セキュリティ問題
  - 対応: 無償修正

□ SLA（Service Level Agreement）
  - 稼働率: 99.5%以上
  - 障害対応時間: 
    - Critical: 1時間以内に対応開始
    - High: 4時間以内
    - Medium: 翌営業日
  - ペナルティ条項

□ 解約条項
  - 途中解約の条件
  - 支払義務の範囲
  - データ・ソースコードの引き渡し

□ 再委託
  - 再委託の可否
  - 事前承認の必要性
  - 再委託先の責任
```

## 5. プロジェクト管理のポイント

### ベンダー管理の体制

```
【顧客側】
プロジェクトオーナー: パートナー1名
├─ プロジェクトマネージャー: 1名（専任）
├─ 業務側代表: アナリスト2名
└─ 技術側代表: 社内エンジニア1名（いれば）

【ベンダー側】
プロジェクトマネージャー: 1名
├─ テクニカルリード: 1名
├─ 開発者: 2-3名
└─ デザイナー: 1名
```

### コミュニケーションルール

```yaml
定例会議:
  - 週次進捗会議（1時間）
    - 進捗報告
    - 課題共有
    - 次週の予定
  
  - 月次レビュー（2時間）
    - マイルストーン確認
    - 予算消化状況
    - スケジュール調整

報告書:
  - 週次報告書（金曜午前）
  - 月次報告書（月末）
  - 障害報告（即座）

コミュニケーションツール:
  - Slack: 日常的なやり取り
  - Zoom: 会議
  - Notion/Confluence: ドキュメント
  - GitHub: コード管理
```

### 品質管理

```markdown
□ レビュー体制
  - コードレビュー: Pull Request必須
  - 設計レビュー: アーキテクチャ決定時
  - セキュリティレビュー: 各フェーズ終了時

□ テスト
  - 単体テスト: カバレッジ 80%以上
  - 統合テスト: 主要シナリオ網羅
  - 受入テスト: 顧客による確認

□ ドキュメント
  - 設計書（アーキテクチャ、API仕様）

  - ユーザーマニュアル
  - ソースコードコメント
```

## 6. よくある失敗パターンと回避策

### 失敗パターン1: 要件定義が曖昧

```
症状:
- 「こういう機能が欲しかった」が後から出る
- 追加費用・スケジュール遅延

回避策:
✓ 要件定義に十分な時間（全体の30%）
✓ プロトタイプで早期に認識合わせ
✓ ユーザーストーリー形式で具体化
✓ 優先順位の明確化（Must/Should/Nice to Have）
```

### 失敗パターン2: コミュニケーション不足

```
症状:
- 認識齟齬
- 問題の発見が遅れる
- 信頼関係の悪化

回避策:
✓ 定期的な対面（オンライン）ミーティング
✓ Slackで日常的にコミュニケーション
✓ 早期の問題共有を奨励（責めない文化）
✓ 顧客側も十分なリソースを割く
```

### 失敗パターン3: スコープクリープ

```
症状:
- 機能がどんどん増える
- 予算・スケジュールオーバー

回避策:
✓ Change Request手続きの確立
✓ 追加機能は次フェーズに
✓ MVPの徹底（最小限で一度リリース）
✓ 定期的なスコープレビュー
```

## まとめ: ベンダー選定の成功要因

### 1. 明確な要件定義
- 何を作るかを具体的に
- 優先順位を明確に
- 成功基準を数値化

### 2. 適切なベンダー選択
- 自社の規模・ニーズに合ったベンダー
- 実績よりも提案内容の質
- 「安さ」だけで選ばない

### 3. 協力的な関係構築
- ベンダーをパートナーとして扱う
- 問題は一緒に解決する
- 成功を共に祝う

### 4. 適切なプロジェクト管理
- 定期的なコミュニケーション
- 早期の問題発見と対応
- 柔軟なスコープ調整

### 5. リスク管理
- 契約で主要リスクをカバー
- エスカレーション手順の確立
- 最悪の場合のプランB準備
