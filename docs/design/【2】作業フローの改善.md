# 情報品質保証プロセス

## 1. 情報の信頼性スコアリング

各情報に自動的に信頼性スコアを付与：

```
信頼性スコア = (情報源の信頼度 × 0.4) + (検証可能性 × 0.3) + (鮮度 × 0.2) + (一貫性 × 0.1)

【情報源の信頼度】
- CONF（監査済み財務諸表、署名済み契約）: 1.0
- INT（経営陣による直接開示、録音あり）: 0.9
- PUB（公式発表、登記情報）: 0.8
- EXT（信頼できるDBのAPI取得値）: 0.6
- EXT（推計値、第三者推定）: 0.4

【検証可能性】
- 第三者による証明書あり: 1.0
- 複数独立ソースで確認: 0.9
- クロスチェック可能: 0.7
- 単一ソースのみ: 0.5
- 検証不可: 0.3

【鮮度】
- 1週間以内: 1.0
- 1ヶ月以内: 0.9
- 3ヶ月以内: 0.7
- 6ヶ月以内: 0.5
- それ以上: 0.3

【一貫性】
- 他情報と完全整合: 1.0
- 軽微な差異のみ: 0.8
- 説明可能な差異: 0.6
- 矛盾あり（要確認）: 0.3
```

## 2. 矛盾検出と解決フロー

```
┌─────────────────┐
│ 情報収集完了    │
└────────┬────────┘
         │
         ▼
┌─────────────────────────┐
│ 自動矛盾検出エンジン        │
│ ・同一指標の値の差異        │
│ ・論理的整合性チェック      │
│ ・時系列の連続性確認        │
└────────┬────────────────┘
         │
         ▼
    ［矛盾検出］
         │
         ├─ No → 次のステップへ
         │
         └─ Yes
              │
              ▼
         ┌──────────────┐
         │ 矛盾の分類     │
         └───┬──────────┘
             │
             ├─【Type A: 説明可能な差異】
             │   例：時点の違い、定義の違い
             │   → 注記を追加して両方保持
             │
             ├─【Type B: データエラーの疑い】
             │   例：桁違い、単位ミス
             │   → 情報源に再確認依頼
             │
             ├─【Type C: 意図的な乖離の可能性】
             │   例：PUBとCONFで大幅乖離
             │   → エスカレーション（要人間判断）
             │
             └─【Type D: 不明な矛盾】
                 → 両方を「要確認」でINT質問票へ
```

## 3. エスカレーション基準

以下の場合は自動処理を停止し、人間の判断を求める：

### 即座にエスカレーション（Red Flag）
- 信頼性スコア < 0.5 の情報が投資判断のクリティカルパスに含まれる
- 同一項目で情報源間の差異が 30% 以上
- 法的・規制上のリスクを示唆する情報
- Deal Breaker 可能性のある情報
- 経営陣の誠実性に関わる矛盾

### 定期レビュー時に確認（Yellow Flag）
- 信頼性スコア 0.5-0.7 の情報がメイン数値に使われている
- 3ヶ月以上古い情報が最新として使われている
- 推計値（EXT）が財務計画の基礎になっている
- 業界常識から大きく外れる数値

### 自動リフレッシュ設定

| 項目カテゴリ | 有効期限 | 自動更新頻度 | 優先度 |
|------------|----------|------------|--------|
| トラクション（ARR, MAU等） | 1ヶ月 | 週次 | 高 |
| 財務状況（現金残高等） | 2週間 | 週次 | 高 |
| チーム情報（人数等） | 3ヶ月 | 月次 | 中 |
| 市場データ（TAM等） | 6ヶ月 | 四半期 | 中 |
| 競合情報 | 3ヶ月 | 月次 | 中 |
| 規制環境 | 随時 | イベント | 高 |

## 5. 検証チェックリスト（自動＋人間）

### 【自動検証】
- [ ] 全ての数値に単位が付与されている
- [ ] 全ての主張に出典が付いている
- [ ] 日付フォーマットが統一されている（ISO 8601）
- [ ] 通貨表記が統一されている
- [ ] 信頼性スコア < 0.5 の情報にwarningフラグ
- [ ] クリティカル項目の充足率 100%
- [ ] 矛盾検出アラートが解決済み

### 【人間による最終確認】
- [ ] 投資テーマと収集された情報が整合している
- [ ] リスク評価が適切（過小評価していない）
- [ ] 競合分析が公平（自社に都合の良いバイアスがない）
- [ ] 財務計画の前提が現実的
- [ ] エグジット想定が楽観的すぎない
- [ ] 開示レベルの分類が適切

## 6. 監査証跡の記録

全ての情報収集・変更・判断について記録：

```json
{
  "audit_log_id": "audit_20251001_001",
  "timestamp": "2025-10-01T14:25:00Z",
  "action": "data_verification",
  "actor": "system|user:tanaka",
  "details": {
    "field": "pre_money_valuation",
    "old_value": null,
    "new_value": 50000000,
    "source": "CONF:term_sheet_v2.pdf",
    "verification_method": "cross_check_with_cap_table",
    "approver": "user:suzuki",
    "confidence_score": 1.0
  }
}
```

この監査証跡により：
- デューデリジェンスの透明性確保
- 規制対応（金融庁等への説明責任）
- 内部統制の証明
- ナレッジの蓄積と学習

### 2-2. タイムライン管理の強化

投資検討プロセスには時間的制約があるため、タイムライン管理を組み込みます：

## 標準タイムライン（初回IC提出まで）

【Week 1: 初期評価】
- Day 1-2: PUB/EXT収集（自動）
- Day 3: 初期スクリーニング（人間判断）
- Day 4-5: CONF依頼、初回MTG設定

【Week 2-3: 詳細調査】
- Day 6-10: CONF資料入手・分析
- Day 11-15: INT実施（経営陣、顧客参照等）
- Day 16-17: ANL完成、ギャップ最終確認

【Week 4: 資料完成】
- Day 18-20: ドラフト作成・内部レビュー
- Day 21-23: 経営陣へのファクトチェック
- Day 24-25: 最終化、IC提出

## タイムライン管理の自動化

- 各タスクに「期限」「依存関係」「オーナー」を設定
- 遅延アラート（期限24時間前、当日）
- ボトルネック検出（待ち状態が48時間以上）
- 代替アクション提案（情報入手不可の場合の次善策）
