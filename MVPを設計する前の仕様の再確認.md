了解しました。MVPを作る前提で、**現状仕様の再確認 → MVP要件の明確化 → 議論が必要な論点（推奨デフォルト付き） → 実装設計（Gemini / GPT + MCPのみ） → 受け入れ基準・品質ガード**の順で整理します。  
このまま設計書として実装に着手できる粒度に落とし込んでいます。

---

## 1) 現状仕様の再確認（要点）

**目的**

- 「なぜファンドはこの案件に投資したのか」を、**ロジカルに一貫して説明できる“決定版”資料**（IC/LP用）として常備する。
    
- 実際に使う機会が限定的でも、**問われた瞬間に即提出可能**な品質・再現性・開示コントロールを担保。
    

**情報ソース方針（タグ）**

- **PUB**：公開情報（公式・PR・登記・メディア・採用 等）→ **自動収集**
    
- **EXT**：外部推定データ（Similarweb, Crunchbase, G2 等）→ **自動収集（API利用範囲）**
    
- **ANL**：分析・計算（TAM/SAM/SOM, LTV/CAC, comps, DCF, リターン）→ **自動計算**（不足セルはPending）
    
- **CONF**：機密内部（Term Sheet, Cap Table, 財務モデル, 契約, KPI定義 等）→ **半自動インジェスト**（人の承認ワンクリック）
    
- **INT**：インタビュー（不足分のみ）→ **最後にギャップ解消**（自動質問票→要点抽出→再計算）
    

**完成物**

- **IC版/LP版**の2系統を**自動生成**（LP版は機微自動マスキング）。
    
- 「結論→根拠→出典」を章ごとに明示し、**全主張にエビデンス（出典/As-of/開示レベル）**を紐付け。

## 2) MVPの非機能要件（受け入れ基準）

- **トレーサビリティ100%**：主要数値・主張の**出典付与率=100%**（URL/ファイル/取得日）。
    
- **再現可能性**：ANLの**式・前提セル**が明記され、第三者が再計算可能。
    
- **整合性**：PUB/EXTとCONF/INTで**数値乖離は検出・記録（反証ログ）**。
    
- **開示コントロール**：IC/LP版の**出し分け自動化**（タグ/レベルでマスク）。
    
- **未充足の可視化**：欠落セルは**`status: pending`**で明示（論点と依存関係が分かる）。
    
- **ドキュメント即時生成**：イベント（案件登録/更新）で**Markdown（IC/LP）**を自動出力。

## 3) 議論が必要なポイント（推奨デフォルト付き）

1. **EXTの対象範囲**
    
    - _論点_：有償API（Crunchbase, Similarweb, data.ai 等）の利用可否・鍵管理。
        
    - _推奨_：MVPは**無料/既存契約内**のAPIに限定。鍵は**MCP側保管**、LLMへ渡さない。
        
2. **文書保管とアクセス権**
    
    - _論点_：原本（CONF）をどこに保存し、誰が見られるか。
        
    - _推奨_：**GDrive（権限最小）**で原本保管、MCPで必要部分だけ抽出→LLMへは**抜粋のみ**。
        
3. **LP版のマスキング規則**
    
    - _論点_：顧客名/単価/契約条項の粒度。
        
    - _推奨_：**社名→セグメント**、**単価→レンジ**、**条項→標準語彙**（例：最低保証=有/無/範囲）。
        
4. **ユニットエコノミクスの定義辞書**
    
    - _論点_：ARR/ARPU/Churn/NDR/LTV/CACの定義ゆれ防止。
        
    - _推奨_：**用語辞書（JSON）**を中枢に置き、抽出/計算/ドキュ生成の**単一参照点**に。
        
5. **価値評価メソッドの既定**
    
    - _論点_：comps優先かDCF優先か、倍率/セクターの選定。
        
    - _推奨_：MVPは**comps第一、DCFは補助**。セクター別の既定マルチプル帯を**リポジトリに固定**。
        
6. **“なぜ今”の証拠**
    
    - _論点_：規制・マクロ・技術トレンドの一次情報源。
        
    - _推奨_：政府/統計・公式発表の**一次ソース優先**（二次メディアは補足）。
        
7. **INTの取得・記録方法**
    
    - _論点_：収録同意、要点抽出の形式（録音/要約テキスト）。
        
    - _推奨_：**要約テキスト投入**を前提（録音はオプション）。同意テンプレを標準化。
        
8. **差分サマリー**
    
    - _論点_：更新検知の粒度と出力位置。
        
    - _推奨_：**章頭に“前回比の変更点”**を箇条書きで自動生成。
        
9. **品質ゲート**
    
    - _論点_：どこで人の承認を必須にするか。
        
    - _推奨_：**CONF抽出の承認**と**最終ドキュ公開前の1回**（2ゲート）。
        
10. **コスト管理**
    
    - _論点_：LLMトークンコスト/外部API費。
        
    - _推奨_：**観測テーブル差分更新**と**セクション別再生成**で最小化。

## 4) データモデル（MVP最小）

**観測テーブル（事実の最小単位）**

```json
{
  "entity_id": "company:acme",
  "section": "business|market|team|deal|kpi",
  "field": "arr|arpu|churn|employee_count|pre_money",
  "value": 123.4,
  "unit": "USD_millions|%",
  "source_tag": "PUB|EXT|CONF|INT|ANL",
  "evidence": "url_or_file_id",
  "as_of": "2025-10-11",
  "disclosure": "IC|LP|LP+NDA|Private",
  "confidence": 0.0
}
```

**依存セル（ANL用）**

```json
{
  "calc_id": "ltv_v1",
  "requires": ["arpu","gross_margin","retention_months"],
  "status": "ready|pending",
  "value": null,
  "notes": "retention_months pending (INT)"
}
```

**ギャップリスト**

```json
{
  "field": "ltv.retention_months",
  "why": "INT必要",
  "suggested_ask": "過去12ヶ月の月次残存率(%)と算出定義を提示ください。",
  "owner": "CFO",
  "priority": "high"
}
```

## 5) アーキテクチャ（Gemini / GPT + MCPのみ）

```
[Trigger: 案件登録/更新]
   ├─ PUB収集Bot（GPT）: MCP:web.search/fetch → 事実抽出(JSON)
   ├─ EXT収集Bot（GPT）: MCP:ext.api(batch)  → 推定値正規化(JSON)
   ├─ 正規化Bot（GPT）   : 重複排除・優先度マージ(CONF>INT>PUB>EXT)
   ├─ ギャップ検出Bot（GPT）: 必須項目の欠落→ギャップ/質問票生成
   ├─ CONFインジェスト（GPT, 半自動）: MCP:gdrive/notion 取込→項目抽出→承認
   ├─ ANL計算エンジン（GPT/ルール）: LTV/CAC, TAM, comps, DCF, シナリオ（穴はpending）
   ├─ INT支援Bot（GPT）: ギャップ→質問票→要点抽出→欠落セル充填
   └─ 資料ビルダー（GPT）: IC/LP版 Markdown 自動生成（差分サマリ・脚注付き）
```

- **MCP**：web.fetch / web.search / gdrive.* / notion.* / ext.*（鍵はMCPにのみ保持）。
    
- **GPT（OpenAI API）**：構造化抽出・正規化・文章生成・JSON/関数呼び出し安定。
    
- **Gemini**：長文PDF/表の読解・比較の補助（出力は同スキーマ）。

## 6) エージェントごとの入出力仕様（要点）

**① PUB収集Bot**

- _IN_：企業名/ドメイン/検索クエリ
    
- _OUT_：観測テーブル配列（数値は期間・分母・通貨必須、出典URL必須）
    

**② EXT収集Bot**

- _IN_：外部APIレスポンス
    
- _OUT_：観測テーブル（`confidence<0.5`、注意書き自動付与）
    

**③ 正規化Bot**

- _IN_：観測テーブル（混在）
    
- _OUT_：優先度マージ済データ＋競合/乖離ログ
    

**④ ギャップ検出Bot**

- _IN_：テンプレ定義＋正規化データ
    
- _OUT_：ギャップリスト（質問文/依頼資料/想定フォーマット/優先度）
    

**⑤ CONFインジェスタ（半自動）**

- _IN_：GDrive/Notionのファイル
    
- _OUT_：観測テーブル（条項→標準語彙、KPI定義抽出）／**承認UI後に確定**
    

**⑥ ANL計算エンジン**

- _IN_：観測テーブル
    
- _OUT_：計算セル（式・前提・出所を併記）／`status: pending` 明示
    

**⑦ INT支援Bot**

- _IN_：ギャップリスト／面談メモ（テキスト）
    
- _OUT_：欠落セルの更新差分（観測テーブル）
    

**⑧ 資料ビルダー**

- _IN_：正規化データ＋ANL＋ポリシー（IC/LP）
    
- _OUT_：Markdown（章：結論→根拠→出典、脚注URL付、差分サマリ）

## 7) プロンプト骨子（抜粋）

- **共通ルール**：
    
    - 推測禁止。未確定は**出力しない** or **`confidence<0.5`**で明示。
        
    - すべてに `source_tag/evidence/as_of/disclosure` を付与。
        
    - 数値は**定義（分母/期間/通貨）**を必ず併記。
        
- **ギャップ生成**：
    
    - テンプレ必須項目に対し未充足→`pending_list[]`
        
    - 質問は**役職別**に割当（CEO/CFO/CTO/CRO/CMO）
        
    - 期待フォーマット（例：CSV, スクショID）を明示。
        
- **LP/IC出し分け**：
    
    - `disclosure != "LP"` は**マスク/レンジ化/匿名化**。
        
    - 章頭に**差分サマリ**を必ず出力。

## 8) 品質ガード & テスト

**自動チェック**

- 出典付与率=100% / As-of必須 / 単位・分母・期間必須
    
- EXTは注意書き必須 / 用語辞書に合致 / 古い指標は警告
    
- 乖離は「反証ログ」に残存
    

**手動ゲート（2か所）**

- CONF抽出の承認（担当：アナリスト/パートナー）
    
- 最終ドキュ公開（担当：IC責任者）
    

**テストデータ**

- ダミー案件×3（業種別：SaaS/Marketplace/Fintech）
    
- 想定エビデンス一式（PR/料金ページ/契約要旨 など）

## 9) バックログ（MVP → 拡張）

**MVP必須**

- 観測テーブル/ギャップ/依存セルのスキーマ実装
    
- PUB/EXT収集・正規化・ギャップ検出のLLMプロンプト
    
- CONFインジェスト（TermSheet/CapTable/財務モデル前提のみ）半自動承認
    
- ANL（LTV/CAC, TAM, comps, シンプルDCF, リターン）
    
- 資料ビルダー（IC/LP Markdown＋差分サマリ・脚注）
    

**拡張（次段）**

- EXT APIの追加（Similarweb, Crunchbase, G2…）
    
- INT支援（質問票→メモ→自動充填のループ）
    
- セクター別マルチプル辞書の高度化／確率重みシナリオ
    
- Google Docs/PDF自動化（MCP経由で可能なら）

## 10) 「なぜ投資したか」を“完璧に”するための芯

- **投資テーマ（3–5行）**：問題の大きさ×解決の独自性×タイミング×チーム×経済性
    
- **エビデンス**：各主張に**一次情報**（PUB/CONF/INT）を直結
    
- **反証と対応**：主要な反論（例：獲得コスト高/更新率不安/規制）と**緩和策**を併記
    
- **リターン経路**：**次ラウンド/エグジットへの道筋**（KPIターゲット・希薄化・BUYER候補）
    
- **価値加算**：ファンドの支援で**何が変わるか**（採用/販路/提携/ガバナンス）
